# GRPO 奖励函数设计

`reward_fn_grpo.py` 为 GRPO 阶段提供规则奖励，由 VeRL 通过 `custom_reward_function` 调用。

## 设计

医疗 MCQ 的奖励由两个维度叠加（满分 1.0）：

| 条件 | 格式奖励 | 正确性奖励 |
|------|---------|-----------|
| 有完整 `<think>` 且推理 >30 字 | 0.2 | 答对得 0.8 |
| 仅有 `<think>` 标签 | 0.1 | 答对得 0.3 |
| 无 think 格式 | 0.0 | 答对得 0.3 |
| 答错 | 同上格式分 | 0.0 |

答案仅从 `</think>` 之后提取，并用负向断言匹配独立的 A/B/C/D，避免把推理过程中提到的字母误判为答案。

## 设计决策

1. **答对但不推理只给 0.3**：拉开"答对且推理"(1.0) 与"直接蹦答案"(0.3) 的差距，逼模型走推理路径，而非跳过 `<think>`。
2. **答案只从 `</think>` 后提取**：防止模型在思考过程中碰巧提到的字母被当作最终答案。
3. **用规则而非神经奖励模型**：MCQ 对错可判定，规则奖励零成本、无 reward hacking 空间。神经奖励模型反而可能被刷分。（开放问答不可判定，故在 DPO 阶段才改用 LLM-as-Judge。）

## 已知局限

本奖励函数刻意保持简单，存在以下局限，列出以备复现者参考：

1. **长度是质量的弱代理**：`len(think_content) > 30` 只衡量长度，不衡量推理质量，理论上可被"医学废话凑字数"刷取格式分。
2. **无推理-答案一致性校验**：奖励只检查"有无 think 块"和"答案对错"，不验证推理是否真正推导出该答案。这是 chain-of-thought **faithfulness（忠实性）** 问题，属领域公开难题，非本实现独有。
3. **不保证临床有效性**：奖励塑造的是"答对 + 使用 think 格式"，推理的医学质量主要继承自 SFT 阶段的 CMExam 真实 COT，而非奖励强制。

## 评测结果对局限的回应

- **reward hacking 未污染泛化**：CMMLU 为训练未见的 held-out 测试集，准确率实测从 39.54% 提升至 65.04%，且经泄漏检测排除（精确重合 0%）。reward 的漏洞未妨碍真实能力提升。
- **推理具备功能性**：`生成式−logit` 指标从 baseline 的 −6.34 反转为 +4.88，说明展开推理实打实帮助答对了 logit 直觉答不对的题——推理至少部分是有功能的，而非纯模式匹配。
- **但不等于忠实诊断**：上述只证明"推理有助答题"，不能证明推理是忠实的临床诊断。本项目为 benchmark/研究性质，未部署于真实诊断场景。

## 改进方向

- 用"包含选项排除分析 / 提及多个选项"等语义代理替代长度阈值
- 引入推理-答案一致性校验（小模型或规则）
- 引入医学事实性检查以提升临床有效性
